## Review: Edge Case Hunter

### Findings

| Location | Trigger Condition | Guard Snippet | Consequence |
|----------|-------------------|---------------|-------------|
| `backend/app/services/inbound_sync.py:186-201` | `_handle_user_created` extracts values from untyped `data` dict via `.get("email", "")`. If webhook `data` contains non-string truthy values (e.g., `{"email": 123, "user_id": true}`), they pass the `if not email or not external_sub:` guard and propagate to `User(email=123, ...)`, crashing at the SQLAlchemy layer. | Validate types: `if not isinstance(email, str) or not isinstance(external_sub, str): return Ok({"status": "skipped", ...})` | [CRASH] |
| `backend/app/services/inbound_sync.py:144-147` | New user flushed (line 131), then `link_repo.create(link)` raises `RepositoryConflictError`. Service returns `Error(Conflict(...))` without rollback. After `IntegrityError`, SQLAlchemy requires rollback before further session use (PEP 249). The session is abandoned in a broken state with an orphan flushed-but-uncommitted user. | After catching `RepositoryConflictError` from link creation, call `await self._user_repo.rollback()`. Or use SAVEPOINTs. | [DATA] |
| `backend/app/repositories/user.py:46` | `UserRepository.create()` calls `await self._session.rollback()` on `IntegrityError`. The session is shared across all three repositories wired in `get_inbound_sync_service()`. Rollback invalidates any prior flushed-but-uncommitted work from sibling repositories. Currently safe (user is first flush), but a latent hazard for future code. | Move `rollback()` to the service layer, or use `session.begin_nested()` (SAVEPOINT). | [DATA] |
| `backend/app/services/inbound_sync.py:203-250` | `_handle_user_updated` updates `User.email` but does not update `IdPLink.external_email`. Over time the link's `external_email` drifts from the actual user email. Any code querying `IdPLink.external_email` returns stale data. | After updating user email, also set `link.external_email = email`. | [DATA] |
| `backend/app/services/inbound_sync.py:252-289` | `_handle_user_deleted` deactivates the user but leaves the `IdPLink` record intact. If the same Descope `user_id` is re-created in Descope, the `(provider_id, external_sub)` unique constraint blocks new link creation. Re-linking requires manual DB cleanup. | On deactivation, remove or soft-delete the IdP link so re-creation is possible. | [DATA] |
| `backend/app/services/inbound_sync.py:186-193` | `_handle_user_created` returns `Ok({"status": "skipped"})` when required fields are missing from webhook `data`. An `Ok` result maps to HTTP 200, signaling to Descope that the event was processed successfully. Descope will not retry. If the field omission is transient (e.g., Descope API field rename), the user creation is permanently lost with no error signal. | Return `Error(ValidationError(...))` instead of `Ok` for missing required fields, so the HTTP response signals failure and Descope retries. | [DATA] |
| `backend/app/services/inbound_sync.py:96-101, 229-232` | Truthiness checks like `if resolved_given:` and `if given_name:` treat empty string as falsy. Name fields can never be cleared to empty via sync -- only overwritten with non-empty values. If Descope sends `given_name: ""` to clear a name, the old value persists silently. | Use `if given_name is not None:` to distinguish "not provided" from "explicitly cleared". | [DEGRADED] |
| `backend/app/services/inbound_sync.py:117-147` | Two concurrent `sync_user_from_flow` requests for the same email but different Descope user IDs. Both `get_by_email` returns `None`, both attempt `user_repo.create`. One wins; the loser gets `Conflict` error and its IdP link is never created. Retry succeeds, but first attempt returns a confusing error. | On `RepositoryConflictError` from user creation, re-fetch by email and proceed to link creation (retry-on-conflict pattern). | [DEGRADED] |
| `backend/app/routers/internal.py:29-30` | `_FLOW_SYNC_SECRET` and `_WEBHOOK_SECRET` are captured at module import time. If env vars are injected after import (late container injection, test patching), cached values stay empty and all requests get 401. The lifespan warning reads `os.getenv` at runtime, so it may see the var while the router's cached value remains empty. | Read secrets inside the dependency functions, or use a settings singleton initialized during lifespan. | [DEGRADED] |
| `backend/app/services/inbound_sync.py:82-85` | `name.split(" ", 1)` assumes Western name ordering (given first, family last). For cultures with family-name-first ordering, the split silently produces incorrect given/family assignments. Only triggers when `given_name` and `family_name` are both absent and `name` is provided. | Document the Western-name-ordering assumption. Consider storing full name in `given_name` when explicit parts are unavailable. | [DEGRADED] |
