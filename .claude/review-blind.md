## Review: Blind Hunter

### MUST FIX

- [`backend/app/middleware/factory.py:73-75`] The entire `/api/internal/` prefix is excluded from JWT authentication via `excluded_prefixes`. This means ANY endpoint registered under that prefix -- now or in the future -- is completely unauthenticated by JWT. The flow sync endpoint (`/api/internal/users/sync`) has zero authentication: no JWT, no HMAC, no API key, nothing. Any network-reachable attacker can POST arbitrary user records into the system, creating users, modifying emails, and linking IdP identities at will. The webhook endpoint at least has HMAC, but the flow sync endpoint is wide open.

- [`backend/app/routers/internal.py:70`] The `FlowSyncRequest` accepts arbitrary `user_id` as a plain string with no validation or authentication. Since the flow sync endpoint has no authentication mechanism whatsoever (JWT is bypassed, and unlike the webhook endpoint there is no HMAC check), an attacker can supply any `user_id` value to create IdP links for arbitrary external subjects, and any `email` to associate them. This is an identity injection vector: an attacker can pre-link a Descope user ID to an email they control, potentially hijacking the account when the real user later authenticates through the same IdP link lookup path.

- [`backend/app/services/inbound_sync.py:93-98`] After a new user is created (line 79, `user_repo.create` which flushes), if the subsequent IdP link creation fails with `RepositoryConflictError` (line 94), the method returns `Error(Conflict(...))` without rolling back. The user was flushed into the session. Meanwhile, `IdPLinkRepository.create()` (line 30 of `idp_link.py`) calls `self._session.rollback()` on IntegrityError, which rolls back the entire transaction including the user creation. But the service still holds a reference to the now-detached/expired user object. The session is left in an inconsistent state. More broadly, the repository calling `rollback()` on a shared session violates transaction ownership -- the service layer should control transaction boundaries, not individual repositories.

### SHOULD FIX

- [`backend/app/routers/internal.py:62-63`] `DESCOPE_WEBHOOK_SECRET` is read from `os.getenv()` on every single webhook request inside the `verify_hmac_signature` dependency. If the env var is unset or empty at startup, the application starts without any warning, giving the false impression webhooks are functional -- every webhook request will silently fail with a 401 "Webhook secret not configured" error. The secret should be validated at startup and injected as a dependency, not fetched per-request from the environment.

- [`backend/app/services/inbound_sync.py:81-84`] In the existing-link update path, `if resolved_given:` and `if resolved_family:` guards mean that empty-string name fields are never written. Since `resolved_given` and `resolved_family` default to `""` when the respective parameters are None, it is impossible to clear a user's name fields via sync. If Descope sends an update with empty/null name fields (intending to clear them), the old values persist silently. The name fields become write-once via this code path.

- [`backend/app/services/inbound_sync.py:79-80`] In the existing-link update path, `user_repo.commit()` on line 80 can raise an exception (e.g., a constraint violation detected at commit time). This exception is unhandled and will propagate as a raw 500. Every other error path in this method returns a typed `Result`, but a commit failure breaks that contract. Same issue on line 97 and line 116-117.

- [`backend/app/repositories/idp_link.py:30`] The `create` method calls `self._session.rollback()` after `IntegrityError`. This repository does not own the session -- it is injected. Rolling back unilaterally invalidates any pending work done by other repositories sharing the same `AsyncSession` (which is the case here: `InboundSyncService` injects the same session into `UserRepository`, `IdPLinkRepository`, and `ProviderRepository`). The service layer should control rollback, not the repository.

- [`backend/app/services/inbound_sync.py:49`] The `email` field is set as an OTel span attribute (`span.set_attribute("user.email", email)`). PII (email addresses) in distributed tracing data can violate GDPR and other privacy regulations, especially if traces are exported to third-party observability platforms. At minimum this should be documented as intentional; ideally email should be hashed or omitted.

- [`backend/app/routers/internal.py:44`] `WebhookPayload.data` is typed as `dict` with no schema validation. Any arbitrary JSON object is accepted and passed through to service handlers which access fields via `.get()` with empty-string defaults. There is no validation that `data` contains the expected fields for each `event_type`. Malformed payloads silently skip processing (returning `Ok({"status": "skipped"})`) rather than signaling an error, making production debugging difficult.

- [`backend/app/services/inbound_sync.py:96-97`] When the IdP link creation succeeds for a newly created user, `commit()` is called via `user_repo.commit()`. But the link was created via `link_repo.create()` which flushed on a different repository object that shares the same session. This works because they share a session, but the commit being called on `user_repo` rather than `link_repo` (or a dedicated unit-of-work) is misleading. If the session scoping ever changes, this implicit coupling breaks silently.

### NITPICK

- [`backend/app/services/inbound_sync.py:64-65`] `resolved_given` and `resolved_family` default to empty string `""` when the respective parameters are None. This means a new user created without any name data gets `given_name=""` and `family_name=""` rather than `None`. Whether "no name provided" and "name explicitly set to empty" should be distinguishable depends on the database schema, but this conflates the two.

- [`backend/tests/e2e/test_internal_endpoints_e2e.py:42`] The assertion `assert resp.status != 401 or "webhook" in resp.text().lower()` is a weak assertion. The `or` clause means the test passes if the response contains the word "webhook" even when the status IS 401. This could mask a real JWT auth rejection if the error body happens to contain "webhook".

- [`backend/app/services/inbound_sync.py:53-55`] The `if not email:` and `if not user_id:` validation checks duplicate validation that Pydantic's `EmailStr` and required `user_id: str` in `FlowSyncRequest` already enforce at the router level. These checks only fire if the service is called directly (bypassing the router), but the redundancy is worth noting.
