## Review: Blind Hunter

### MUST FIX

- **[`backend/app/repositories/user.py:46,68` + `backend/app/repositories/idp_link.py:46-47`] Inconsistent rollback behaviour across repositories sharing a session.** `UserRepository.create()` (line 46) and `UserRepository.update()` (line 68) call `self._session.rollback()` on `IntegrityError`. `IdPLinkRepository.create()` (line 46-47) does NOT rollback -- it re-raises `RepositoryConflictError` and explicitly documents "Does NOT rollback -- the caller (service layer) owns the transaction." This asymmetry is dangerous in `sync_user_from_flow` where both repositories share the same `AsyncSession`. Scenario: in the new-user path, `user_repo.create(new_user)` succeeds (flushed to session), then `link_repo.create(link)` raises `RepositoryConflictError` without rolling back. The service returns `Error(Conflict(...))`, but the flushed-but-uncommitted user remains in the session. If the session is somehow reused (e.g., FastAPI's dependency scoping doesn't end it immediately), the orphaned user could be committed by a subsequent operation. Conversely, if `user_repo.create()` itself hits an `IntegrityError`, it rolls back the entire shared session -- potentially invalidating any work flushed by other repositories in a multi-step operation. The repositories should either all rollback or none should; the current mix violates the transactional contract.

- **[`backend/app/routers/internal.py:29`] `_FLOW_SYNC_SECRET` is captured at module import time, but Python modules are imported once and cached.** If the `DESCOPE_FLOW_SYNC_SECRET` environment variable is not set when the module is first imported (e.g., during test collection or when FastAPI discovers routers), `_FLOW_SYNC_SECRET` is permanently `""` for the process lifetime. Setting the env var later (e.g., via a late-loading secrets manager) has no effect. Every flow sync request will be rejected with 401 "Flow sync secret not configured" and there is no log message at the point of import to indicate the secret was empty. The startup warning in `_warn_missing_secrets()` (main.py:41) only logs -- it does not prevent the application from serving requests. Combined with the fact that `_FLOW_SYNC_SECRET` is a module-level variable that tests mutate directly (test_internal_router.py:83,97,119,137,159), the production secret-loading path is never actually tested by the unit tests.

### SHOULD FIX

- **[`backend/app/services/inbound_sync.py:96-101,229-232`] Name fields cannot be cleared through sync.** In the existing-link update path of `sync_user_from_flow` (lines 98-101), `if resolved_given:` and `if resolved_family:` use truthiness checks. Since `resolved_given` defaults to `""` when `given_name` is `None`, a falsy value is never written. The same pattern appears in `_handle_user_updated` (lines 229-232). If Descope sends an update with empty/null name fields (intending to clear a user's name), the old values persist silently. Name fields become write-once through both code paths.

- **[`backend/app/services/inbound_sync.py:223-225`] Webhook `_handle_user_updated` accepts email from untyped `data` dict without format validation.** The flow sync path validates email via Pydantic `EmailStr` at the router layer. The webhook path reads `email = data.get("email")` and writes it directly to `user.email` with only a truthiness check (`if email:`). A malformed email string from a corrupted or malicious webhook payload would be stored in the database. While the unique constraint prevents duplicates, there is no format validation. Two code paths that write the same field have inconsistent validation levels.

- **[`backend/app/services/inbound_sync.py:108-112,149-153,243-247,283-286`] Commit failures are caught with bare `except Exception` and returned as `Conflict` errors.** A `DisconnectionError`, `OperationalError`, or timeout during commit will be returned to the caller as a 409 Conflict (via `result_to_response` mapping of `Conflict` to HTTP 409). This masks infrastructure failures as business logic errors. Operators monitoring for 409s to detect duplicate-data issues will also see database outages in the same bucket, and clients that retry on 409 may retry unnecessarily. The error type should distinguish "conflict" from "infrastructure failure."

- **[`backend/app/services/inbound_sync.py:195-201`] `_handle_user_created` delegates to `sync_user_from_flow` but the email from `data.get("email", "")` bypasses Pydantic `EmailStr` validation.** A `user.created` webhook with `"email": "not-a-valid-email"` would pass the `if not email:` truthiness check and create a canonical user with an invalid email address. The flow sync router path validates email format; the webhook path does not.

- **[`backend/app/routers/internal.py:49-50`] `WebhookPayload.data` is typed as bare `dict` with no schema validation.** Any arbitrary JSON object is accepted and passed to service handlers, which access fields via `.get()` with empty-string defaults. There is no validation that `data` contains the expected fields for a given `event_type`. Malformed payloads silently return `Ok({"status": "skipped"})`, making production debugging difficult -- a payload structure change from Descope would silently drop all events.

- **[`backend/app/models/identity/provider.py:27`] `Provider.type` column has `index=True` but not `unique=True`.** The schema permits multiple providers with the same `ProviderType`. `ProviderRepository.get_by_type()` uses `scalars().first()` to silently pick one, but if two `descope` providers exist with different configurations (e.g., different `issuer_url`), the system non-deterministically selects one based on database row ordering. A unique constraint on `(type, active)` or at least `type` would prevent this data integrity issue at the schema level rather than relying on operational discipline.

### NITPICK

- **[`backend/app/services/inbound_sync.py:80-81`] `resolved_given = given_name or ""` and `resolved_family = family_name or ""` conflate `None` and `""`.** Whether "no name provided" and "name explicitly set to empty" should be distinguishable depends on domain semantics. The `User.given_name` column defaults to `""`, so this is consistent with the schema, but the semantic distinction is permanently lost.

- **[`backend/app/services/inbound_sync.py:82-85`] Name splitting via `name.split(" ", 1)` is simplistic for international names.** Names in family-name-first cultures (East Asian, Hungarian) will be split with given/family reversed. This is a known hard problem, but worth documenting as a known limitation.

- **[`backend/tests/unit/test_internal_router.py:155-161`] `_set_flow_secret` fixture is `autouse=True` and mutates a module-level variable.** This means every test in the module gets `_FLOW_SYNC_SECRET = FLOW_SECRET`, including tests that explicitly test the "unconfigured secret" path (line 133). Those tests work by setting the variable before the autouse fixture runs, relying on pytest fixture ordering. This is fragile -- if fixture ordering changes, the "unconfigured" tests would silently pass with the wrong secret state.

- **[`backend/app/services/inbound_sync.py:68-72`] Redundant validation checks.** `if not email:` and `if not user_id:` duplicate validation that Pydantic's `EmailStr` (which rejects empty strings) and required `user_id: str` in `FlowSyncRequest` already enforce at the router level. These only fire when the service is called directly (e.g., from `_handle_user_created`), which does justify their existence for the webhook path -- but this should be documented as intentional defense-in-depth rather than appearing as accidental redundancy.
